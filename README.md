> **Upload a resume → get a production-ready portfolio website in seconds. 100% browser-based, zero backend.**

| Feature | Details |
|---|---|
| 🤖 **Multi-Provider AI** | Groq (Llama 3.3 70B), Google Gemini 2.0 Flash, OpenAI GPT-4o Mini, OpenRouter |
| 🔑 **Key Pool Rotation** | Add multiple API keys — auto-rotates on rate-limit, never blocks you |
| 📄 **Resume Parsing** | PDF.js + Mammoth.js — extracts text 100% in the browser, zero server uploads |
| 🎨 **Design Presets** | 6 distinct aesthetics: Dark Minimal, Editorial, Terminal, Glassmorphism, Bold, Clean |
| 📦 **One-Click Export** | Downloads a clean ZIP with separate `index.html`, `style.css`, `script.js` |
| 🧪 **36 Unit Tests** | Vitest test suite — parsers, validators, file handling, error classes |
| 🚀 **Auto-Deploy** | GitHub Actions CI/CD → GitHub Pages on every `main` push |
| 🔒 **Privacy First** | Keys live in JS memory only — never `localStorage`, never a server |

<details>
<summary><b>🏗️ Architecture Deep-Dive (click to expand)</b></summary>

```
portfolioforge/
│
├── index.html                # App shell: semantic HTML + ARIA roles, zero inline styles
│
├── src/
│   ├── style.css             # Full design system via CSS custom properties
│   │   ├── :root {}          # 20+ variables: colors, fonts, radius, transitions
│   │   ├── animations        # sweep, blink, reveal-up, fade-in, badge-in @keyframes
│   │   └── responsive        # 768px + 480px breakpoints, mobile-first approach
│   │
│   └── app.js                # All logic — vanilla ES2022, zero runtime dependencies
│       ├── STATE {}           # Single source of truth: provider, keyPool[], html/css/js
│       ├── PROVIDERS {}       # Per-provider config: endpoint, model, validate(), helpHTML
│       ├── Key Pool Engine    # addKeyToPool(), markKeyRateLimited(), getNextKey()
│       ├── callWithKeyRotation()   # Round-robin over keys → multi-round backoff
│       ├── extractPDF()       # PDF.js page-by-page Y-coordinate text reconstruction
│       ├── extractDOCX()      # Mammoth.js raw text extraction
│       ├── cleanResume()      # Normalise whitespace, hard-cap at 1800 chars
│       ├── buildSystem()      # ~400-token system prompt (fits Groq free tier)
│       ├── callOpenAICompat() # Handles Groq / OpenAI / OpenRouter with typed errors
│       ├── callGemini()       # Gemini-specific request format + error mapping
│       ├── parseAI()          # 4-strategy parser: delimiters → fences → normalised → fallback
│       └── renderPreview()    # Sandboxed srcdoc iframe with injected CSS + JS
│
├── tests/app.test.js         # 36 tests across 6 describe blocks
│   ├── Provider validation    # Prefix + length for all 4 providers
│   ├── parseAI()              # All strategies, error cases, tiny HTML rejection
│   ├── fmtBytes()             # B / KB / MB edge cases incl. zero and exact boundaries
│   ├── Prompt presets         # Key existence and type checks
│   ├── File validation        # Type, size, uppercase extensions, exact 5MB boundary
│   └── Custom errors          # RateLimitError, ParseError, AllKeysError name + instanceof
│
└── .github/workflows/ci.yml
    ├── test     # Node 20, npm ci (requires package-lock.json), vitest run
    ├── validate # html-validate (non-blocking, reports only)
    └── deploy   # Independent of test job — Pages deploys regardless
```

**Key Design Decisions:**

- **No bundler** — plain HTML/CSS/JS. Zero build step, instant `git push` deploys. The only "build" is writing the files.
- **No localStorage for secrets** — API keys live in `const STATE = {}` only, garbage-collected on tab close. This is deliberate, not an oversight.
- **Retry architecture** — `callWithKeyRotation()` implements a round-robin loop over all valid keys. Rate-limited keys are flagged and skipped. After exhausting all keys in one round, it waits `round * 15` seconds before the next round (max 3 rounds).
- **Token budget** — `cleanResume()` normalises whitespace and caps at 1,800 chars (~450 tokens). The system prompt is ~400 tokens. Total request stays under Groq's free-tier per-request limit even with a verbose resume.
- **4-strategy parser** — LLMs occasionally ignore output format instructions. The parser tries: exact delimiters → markdown code fences → whitespace-normalised delimiters → raw HTML tag detection. This makes generation succeed even when the model partially drifts.
- **Accessibility** — ARIA roles, `aria-live` regions, `aria-checked` on radio groups, `role="status"` on file bar, keyboard navigation on upload zone (Enter/Space), `focus-visible` outline.

## 📐 Engineering Philosophy

<div align="center">

```
"Ship it, then improve it.
 Tests first on anything that has state.
 If it needs a server, question why.
 If it needs a framework, question harder."
```

</div>

| Principle | Practice |
|:---|:---|
| 🎯 **Simplicity over cleverness** | Readable 10-line function over a smart 3-liner |
| 🚢 **Ship early, iterate fast** | Working v1 beats perfect v0 every time |
| 🧪 **Test behaviour, not implementation** | Tests describe *what*, not *how* |
| 🔒 **Security by default** | Keys in memory only; HTTPS always; no implicit trust |
| 📦 **Zero-dependency where possible** | Vanilla JS over a framework for focused tools |
| 📝 **Docs are code** | README, inline comments, and types are first-class citizens |
